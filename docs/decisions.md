# 技術選定・意思決定記録

このドキュメントは「何を選んだか」ではなく「**なぜ**それを選んだか」を記録する。`architecture.md` の内容（構成そのもの）と対になる。

## 1. フロントエンド: Vite + Vanilla TypeScript

**選定**: React等のフレームワークを使わず、Vite + Vanilla TypeScript

**理由**:
- 最重要要件が「速度最優先」（撮影から解析完了まで体感で即座）であり、UIは単一画面・状態遷移がほぼないため、Reactの仮想DOM・状態管理のオーバーヘッドを払う理由がない
- カメラプレビュー・Canvas前処理・ONNX Runtime WebはいずれもDOM/WebAPIを直接叩く処理で、フレームワークの抽象化がむしろ間に挟まる
- 依存が少ないほどバンドルサイズが小さくなり、初回ロード・オフライン起動が速い（PWAの要件と合致）

**不採用にした選択肢**: Vite + React + TypeScript — 状態管理やコンポーネント分割が必要になるほど画面が複雑化した場合に再検討する

## 2. AI推論言語: TypeScript（Rust/C不採用）

**選定**: ブラウザ側の実装言語はTypeScriptのまま。Rust/C（Wasm化）は導入しない

**理由**:
- 推論のホットパスはONNX Runtime Web自体（内部はC++実装のWasm/WebGLバックエンド）が担っており、呼び出し側言語を変えても推論速度そのものは変わらない
- 画像前処理（リサイズ・正規化）に使うCanvas APIもブラウザネイティブ実装（C++）で、Wasm化するとJS-Wasm間のシリアライズが余分に乗り、単純な場合はむしろ遅くなり得る
- Rust/Cを混ぜるとwasm-bindgen/Emscriptenのビルドパイプラインが増え、devbox環境・CI・デバッグの複雑さが跳ね上がる。今回のUIロジックの薄さに対して見合わない
- 速度向上の本丸は「モデルアーキテクチャの軽量化（MobileNetV3等）」と「ONNX変換の最適化」であり、これは言語選択より効くレイヤー

**再検討の条件**: 文字領域検出ロジックや前処理が実測でボトルネックになった場合、その部分のみ限定的にWasm化（Rust）を検討する

## 3. Node.js: v22（v20不採用）

**選定**: devbox.jsonで `nodejs@22` を指定

**理由**:
- 当初 `nodejs@20` を指定したが、Nixのビルド時に `nodejs-20.20.2` が既知の脆弱性により insecure package としてブロックされた（`NIXPKGS_ALLOW_INSECURE=1` で強制許可も可能だが、セキュリティ例外を積む理由がないため回避）
- v22はLTSで、Viteの動作要件も満たすためv22に変更

## 4. パッケージマネージャー: pnpm（npm不採用）

**選定**: pnpm

**理由**:
- インストール速度・ディスク効率で優位（ハードリンクベースのストア）。「速度重視」の開発体験にも合致
- devboxとの相性・最近のテンプレートプロジェクトでの採用実績を優先

## 5. PWA構成: 単一画面・カメラ即起動

**選定**: 画面遷移なしの単一画面。PWAのURLを開いたら即座に `getUserMedia` でカメラを起動し、下部固定エリアに翻訳結果を表示

**理由**:
- ユーザーの利用シーンが「紙に印刷された文字をその場で翻訳する」という一点に絞られており、設定画面や履歴画面などを挟むと体感速度が落ちる
- 画面遷移を挟まないことで、状態管理・ルーティングライブラリが不要になり、Vanilla TypeScript構成とも整合する

**未確定事項**: キャプチャ方式（撮影ボタン式か、常時解析ループの自動検出式か）は `requirements.md` 8章で未確定のまま。自動検出にする場合は常時推論のバッテリー消費とのトレードオフを検証してから決定する

## 6. Lint/Format: ESLint (typescript-eslint) + Prettier

**選定**: ESLint flat config（`eslint.config.js`）+ Prettier、`eslint-config-prettier` で競合ルールを無効化

**理由**:
- ユーザーのグローバル規約（`any`禁止・`unknown`使用、厳密な型付け）を機械的に強制するため、typescript-eslintのrecommendedルールセットを採用
- フォーマットとlintのルールが競合しないよう、ESLint側のスタイルルールはPrettierに譲る構成にした
